# Using Skills

This is a mandatory operating protocol. It is not advice. It is not optional.

## The 1% Rule

Before taking ANY action — responding, dispatching, analyzing, or delivering — check which skills apply to your current task. If there is even a **1% chance** a skill is relevant, you MUST invoke it BEFORE acting.

## How to Check

1. Read the skill names and triggers loaded in your context
2. Compare against what you are about to do
3. If a skill might apply: follow its rules exactly
4. Announce which skill you are using: "Invoking [skill-name] because [reason]."

## Red Flags — You Are Avoiding the Process

If you catch yourself thinking any of these, it is a signal TO follow the skill, not a reason to skip it:

- "This is too simple for a formal process" → Invoke the skill anyway.
- "I already know what to do" → The skill catches what intuition misses.
- "I'll just quickly do this one thing first" → That is the avoidance pattern. Stop. Check skills.
- "The user wants speed, not process" → Speed without process produces rework. Process IS speed.
- "I'll apply the skill next time" → There is no next time. Apply it now.

## Priority

**Process skills** (scope-assessment, brainstorming, quality-gating) take priority — they determine your approach.
**Execution skills** (delegation-dispatch, multi-phase-orchestration) come second — they guide how you work.
**Verification skills** (verification-before-completion) come last — they validate your output.

## Non-Negotiable

Skipping a skill check is never acceptable. Every turn begins with an implicit skill applicability assessment. If no skill applies, proceed normally. If any skill applies, follow it before doing anything else.
